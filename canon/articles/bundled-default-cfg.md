# Bundled Default Cfg

> **What this answers.** What `ptxprint.cfg` does the Container ship with? What does it produce? When does an agent need to override it?
>
> **What this is.** The source of truth for the cfg the Container stages as the default when the caller's payload provides no `shared/ptxprint/Default/ptxprint.cfg` entry. The fenced INI block at the bottom is the literal file content. Container build copies from this article; the article is canonical, the build artifact is derived.

---

## Why this article exists

Per `klappy://canon/principles/dry-canon-says-it-once`, the bundled cfg has exactly one home. That home is this article. The Container build pulls from here. If the upstream PTXprint team sends a new default, the cfg in this article is updated, the Container rebuilds, and every Phase-0 caller picks up the new defaults on the next render.

If the cfg lived in the Container repo only, agents reasoning about "what does the default produce" would have to inspect Container internals — opaque to the canon-driven retrieval pattern. By living here, the cfg is `docs`-discoverable.

## What this cfg produces

When a caller submits the smallest possible payload (`{books, sources}`) the Container stages this cfg at the project's Default config path, runs PTXprint, and emits:

- **Two-column justified** A5 layout (148mm × 210mm)
- **12pt margins**, 18pt top, 10.5pt bottom
- **Charis** (regular / bold / italic / bold-italic) — resolved via the Container's bundled `fonts-sil-charis` package
- **Drop cap** on chapter 1, regular chapter numbers thereafter
- **Section heads in italic, parallel-passage references** below them in italic
- **Footnotes** at column foot with autogenerated alphabetic callers (`a`, `b`, `c`, …)
- **Cross-references** at column foot, separately ruled
- **Mirror-layout running header**: book name + chapter on outer, page number on inner
- **No front matter, no figures, no cover, no decorations**

Empirically validated: BSB John (one chapter through 21) renders to a 37-page PDF/X-4 in ~15 seconds container time, 0 errors, 0 overfull boxes. See `klappy://canon/handoffs/bundle-default-cfg-handoff.md` for the smoke-test contract.

## Where this cfg comes from

Provided by the upstream PTXprint team on 2026-04-30 as the cfg that ships with PTXprint by default in a real working BSB project. PTXprint version stamp embedded in the cfg: `2.9.9-gc5521963`. Cfg schema version: `2.24`.

The upstream team's framing: this is the default that will be in PTXprint. Adopting it as the Container's bundled fallback is therefore a forward-compatibility move, not a custom opinion. When upstream ships a new release with a different default, this article is updated to match.

## Known characteristics worth knowing as an agent

These aren't bugs — they're knowable behaviors of the bundle:

1. **No `[vars]` section.** The cfg has no `[vars]` block. The PDF metadata `Title` field shows as `[Unknown]`. To set a real title, callers either add `[vars] maintitle = ...` via a full `config_files` cfg replacement, or override at run-time via `define = {"maintitle": "..."}` (preferred for one-shot use). See `klappy://canon/articles/progressive-customization`.

2. **Hard-coded `book = JHN, bookscope = single`.** This cfg targets John specifically. PTXprint reads the payload's `books` array as the actual run scope, but the cfg's `book` field surfaces in metadata. For other books, callers either (a) ignore the discrepancy (cosmetic — not a render error), or (b) override via `define = {"book": "ROM"}` (PTXprint widget identifier). For multi-book renders, override `bookscope` and `booklist` via `define` or replace the cfg.

3. **Underfill on light-content pages.** Tight typography (`linespacing 11.642`, `fontfactor 9.5`) leaves visible empty space at the bottom of columns where text doesn't fill. On BSB John this hits ~59% of pages (22 of 37). The PDF is valid; the layout is conservative. The canonical fix is `mode: autofill` (deferred). Interim mitigations available via `define`: `s_linespacing`, `c_preventwidows`, `c_preventorphans`.

4. **License-vs-colophon contradiction.** `[project] colophontext` says "dedicated to the public domain"; `[project] license` says `CC BY-NC-ND 4.0`. The colophon text wins in the rendered PDF (when colophon material is in scope). Callers publishing under their own license should override `colophontext` and `license` via a `config_files` replacement. Surfaces only when front/back matter is enabled.

5. **`[import]` section is for the import-settings GUI workflow, not for parent-config inheritance.** PTXprint's GUI uses `[import]` to track "where I last imported settings from"; it is **not** the same as the `[import] config = Parent` inheritance pattern documented in `klappy://canon/articles/config-inheritance-and-overrides`. Treat the section as opaque metadata; do not edit it to attempt inheritance.

## Override path summary

| What you want | How |
|---|---|
| Use the bundle as-is | Send `{books, sources}` only — empty `config_files` |
| Change one knob | Add `define = {"<widget_id>": "<value>"}` |
| Change many knobs / replace cfg | Add `config_files["shared/ptxprint/Default/ptxprint.cfg"]` with your own cfg |
| Different fonts | Add `fonts: [...]` array with URL+sha256 entries |
| Custom stylesheet | Add `config_files["shared/ptxprint/Default/ptxprint.sty"]` |
| Add figures | Add `figures: [...]` array + a piclist in `config_files` |
| Front matter | Add `config_files["shared/ptxprint/Default/FRTlocal.sfm"]` and override `iffrontmatter` |

The full ladder lives in `klappy://canon/articles/progressive-customization`.

## Maintenance

When upstream provides an updated default cfg:

1. Replace the fenced INI block at the bottom of this article with the new content.
2. Bump `asset_version` and `ptxprint_version_origin` / `cfg_schema_version` in this article's frontmatter.
3. Trigger a Container rebuild. The Container's build step reads this article's INI block at build time (or the canon-as-asset feed) and stages it at the install path.
4. Run the smoke test in `klappy://canon/handoffs/bundle-default-cfg-handoff.md`.

The cfg is content-addressed by the Container build; same article SHA → same Container behavior.
